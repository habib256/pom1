## ⚡ gen2c — perf (profilé 2026-06-16)

> Profilage PC-histogramme du Snake GEN2 HGR (`dev/projects/gen2_snake_telemetry/`) en cours de jeu, throttle minimisé pour exposer le *travail* derrière la temporisation. Verdict : à part le `throttle` (volontaire, 32 %), le coût est la **glue d'appel + helpers arithmétiques cc65 (~31 %)** étalée sur le chemin par-frame (`decsp/incsp/popax/popptr1`, `aslax3` pour ×8, `tosaddax`, `bool*`) — pas une fonction chaude. `gen2_pixrect_asm` (13 %, dessin cellules) est **déjà asm et quasi optimal**. `tick` 7 %, `gen2_hgr_putu` 6 %. Les deux leviers réels (tous deux **encore ouverts** — vérifié 2026-06-16 : pas de `gen2_hgr_cell`, le corps Snake est toujours reconstruit en O(longueur)) :

- [ ] **Blitter cellule asm dédié `gen2_hgr_cell(cx, cy, set)`** `[S · solid]` — un seul appel asm prenant les coords **en cases** (pas en pixels) : il fait le `cx*8`/`cy*8` (×8 = shift) **et** le remplissage 6×6 en interne, supprimant les wrappers C `draw_cell`/`erase_cell` **+** le wrapper C de `gen2_hgr_fill_pixrect` **+** les `jsr aslax3` par cellule (2 cellules/frame). C'est le plus gros gain sur les 31 % de glue cc65. À ajouter dans `dev/lib/gen2c` (`gen2.c`/`gen2.h`/`gen2_blit.s`) à côté de `gen2_pixrect_asm` ; Snake câble `draw_cell`/`erase_cell` dessus.
- [ ] **Corps du serpent en ring-buffer** `[S · solid]` — **algorithmique (pas d'asm)** : la boucle `for(i=slen-1;i>0;--i){sx[i]=sx[i-1];sy[i]=sy[i-1];}` dans `tick()` est O(longueur) **chaque tick** et grossit avec le serpent (gros du churn array/helpers cc65). La remplacer par des index head/tail circulaires → O(1) par tick. Touche `tick()` + `draw_cell` boucle + collision dans `GEN2Snake.c` uniquement.

> NB : `gen2_pixrect_asm` (13 %) déjà optimal — ne pas y toucher. Le `throttle` (32 %) est la temporisation voulue (contrôle de vitesse), pas un goulot.

## 🧩 Factoriser les libs graphiques GEN2 HGR ↔ TMS9918 (3 axes)

> **Contexte** (analyse 2026-06-16) : les libs C/asm des deux cartes graphiques sont conceptuellement parallèles mais **dupliquées et dispersées**. Trois couches C : base neutre `dev/lib/apple1c/apple1io.h` (partagée) · runtime GEN2 `dev/lib/gen2c/gen2.h` (riche, in-repo) · runtime TMS9918 `dev/apple1-videocard-lib/lib/*` (Nino Porcino, **hors `dev/lib`**). **Déjà factorisé** : les *sprites* (`tools/build_hgr_sprites.py`, master TMS → miroir HGR) et la *fonte BB côté HGR* (`emit_bbfont.py` : fontbb.s → bbfont_cp437 → gen2_bbfont). La **géométrie vectorielle** (ligne/cercle/rect/ellipse) et la **conversion entier→ASCII** sont identiques des deux côtés mais écrites 2-3 fois. Plan en 3 axes ; règle Parmigiani « une carte à la fois » → **backend résolu au lien**, pas de pointeur de fonction.

### Axe 1 — couche géométrie/nombres partagée (`dev/lib/gfx/`)

> **✅ Couche posée + recâblage destructif GEN2/TMS + intégration build livrés → [`../CHANGELOG.md`](../CHANGELOG.md)** (`gfx.h` + `gfx_draw.c`/`gfx_num.c` + backends GEN2/TMS, archives `ar65` à liaison-à-la-demande ; les deux cartes forwardent vers `gfx_*`, fast-fill TMS inclus). Reste la régression visuelle pixel-exacte + une optimisation taille optionnelle.

### Axe 3 — façade `gfx_*` du tronc commun (cadrée, PAS une fusion totale)

> Après axes 1-2, exposer un header neutre pour le tronc commun (pixel/texte/nombres/vecteurs) afin qu'un jeu se compile pour les deux cartes par simple choix de backend au lien. **Limites** : la règle « une carte à la fois » fait que la façade n'apporte que de la portabilité **source** ; et il faut **conserver** les API spécifiques (sprites HW + collision TMS ; double-buffer + couleur NTSC + blit GEN2) — ne PAS niveler par le bas.

- [ ] **Façade tronc commun texte/curseur** `[M · solid]` — étendre `gfx.h` au texte/nombres positionnés via un modèle commun de curseur de cellule, backends `gfx_text`/`gfx_putc` par carte (police graphique GEN2 vs cellule 8px TMS). Garder les API riches spécifiques intactes.
