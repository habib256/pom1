  - **Cause n°1 confirmée (Snake + Sokoban) — power-on VRAM trop propre.** `TMS9918::reset()` (`src/TMS9918.cpp:134-152`) a deux profils : défaut **`vramNoiseOnReset = false`** → motif bistable `$FF/$00` (`vram[i]=(i&1)?0x00:0xFF`) qui met les octets Y non initialisés de la SAT à `$FF` → sprites parasites **hors écran**, donc un jeu qui zappe le fill défensif rend bien ; **`vramNoiseOnReset = true`** → vrai bruit `mt19937`, et le commentaire dit lui-même *« what the warm P-LAB DRAM actually shows on cold boot. Côté jeux : **Snake** (`sketchs/tms9918/game_snake/TMS_Snake.asm:866-874`) + **Sokoban** écrivent seulement `SAT[0].Y=$D0` et **omettent** `SAT[1..127]=$D1` (cf. `TMS9918-SPRITE_INIT.md §4.2`) ; Plasma/Life/Galaga font le fill complet (référence).
  - **Cause n°2 — workflow multi-agents + piège RAM (2026-06-29).** Le `[TMS9918 DROP]` est **réel** (write abandonné avant `vram[]`), candidat éliminé. Le workflow a isolé **deux** mécanismes distincts pour les jeux à init SAT complète :

Pinned par `basic_native_run` (entier **et** flottant). **Chiffres** (natif vs même source sur l'interpréteur, sortie identique) : boucle de calcul entière **~22×**, tracé de lignes entier **~4.5×**, **parabole flottante ~2.0×** (le plafond honnête du FP-bound : la binary32 n'est pas plus rapide que la FP ROM, le gain vient du retrait de l'overhead d'interprétation). **Auto-précision + dead-stripping + diagnostics livrés** : `FpMode::Auto` choisit le plus petit type suffisant (entier sauf si fraction `/` ou décimale → binary32 ; sans float → runtime float non lié) ; le runtime est *dead-strippé* via `-D RT_xxx` (routines + tables 560 o inutilisées absentes du binaire — `X=5+2:X=X+1` **1695→89 o**, `PRINT`+`FOR` **1746→265 o**) ; erreurs claires nommant la **ligne Applesoft exacte** + validation des cibles `GOTO/GOSUB/THEN` à la compilation. **Phase 2c livrée — `3DHat.apf` compile et tourne en natif sur GEN2 *et* TMS9918** : `SIN`/`SQR`/`INT` ajoutés (`fp_sin` réduction 2π + Taylor 4 termes, `fp_sqrt` Newton 5 itér., `fp_int` troncature ; chacun *feature-gated* `-D FP_SIN/SQRT/INT`, vérifiés vs `sinf`/`sqrtf` hôte — 5736 cas), `AND`/`OR` flottants corrigés, **`HCOLOR=0` efface** (`rt_plot` pen-aware → masquage des lignes cachées du HAT), et un **peephole** (`optimizePeephole`) qui fusionne les allers-retours par temp (~640 o gagnés) pour faire tenir 3DHat dans la fenêtre GEN2 `$0300–$1FFF` (cfg : BSS déplacé en `$0200`, code jusqu'au framebuffer `$2000`). Le sombrero se dessine avec masquage des lignes cachées, sans ROM. Pinned par `basic_native_run` (cas 3DHat natif-only) + `basic_native_bench` (table taille+vitesse par type de programme : int-arith **21×**, raster **14×**, lignes **2.4×**, parabole **2.0×**, transcendantal **1.4×** ; binaires 354 o–7157 o). 3DHat natif est *compute-heavy* (≈20 000 évals transcendantales + lignes d'effacement) donc lent à terminer, mais plus rapide que l'interpréteur. **Lo-res livré sur les deux cartes — `RodColor.apf` compile + tourne en natif sur GEN2 et TMS9918.** `GR/GR2`, `COLOR=`, `PLOT`, `HLIN/VLIN … AT`, `TEXT`, `HOME` + `ONERR GOTO` natifs. GEN2 : framebuffer 40×48 page `$0800` ; TMS9918 : **Multicolor** 64×48 dans la VRAM privée (routines autonomes dans `basicrt_tms.s`, ne lie que `tms9918_pad12`, pas `tms9918m2.o`). **Deux détails de déploiement** rendent un vrai programme lo-res viable : (1) **GEN2 lo-res charge à `$0C00`** (`basicc_native_gen2_lores.cfg`), au-dessus des deux pages lo-res, sinon un programme à `$0300` s'écrase lui-même dès que `GR`/`PLOT` peint la page 2 `$0800` (HGR reste à `$0300`) ; (2) **le preset TMS9918 (CodeTank) est le dual-bank Parmigiani 8 Ko** (RAM seulement `$0000-$0FFF`) — le déploiement Bench le relâche à 16 Ko avant de lancer un binaire natif (l'image autonome possède la fenêtre contiguë `$0300-$1FFF` du cfg ; sans relâche, tout au-delà de `$0FFF` lit `$FF` en OOR strict et plante). Pinned par `bench_basic_native_lores_smoke` (GEN2, charge à `$0C00`) + `bench_basic_native_lores_tms_smoke` (TMS, inspecte la VRAM via `copySnapshot` ; vérifie aussi RodColor flottant : peint à 16 Ko, vide à 8 Ko). **`COS` livré** (`fp_cos` = `sin(x+π/2)`, réutilise le cœur Taylor de `fp_sin`, *feature-gated* `-D FP_COS`, vérifié vs `cosf` hôte dans `basic_float_runtime`) — `Snowflake.apf` (Kevin Riordan 1985 : `HGR2`, `SIN`/`COS`/`INT`, lignes `HPLOT`) compile + tourne en natif sur TMS9918 (7146 o ; trop gros de ~455 o pour la fenêtre HGR GEN2 `$0300-$1FFF`). **Sources `.apf` lisibles** : les listings Applesoft sans espaces (`FORL=1TOINT`, `80to0`) sont édités pour ré-introduire les espaces — le compilateur natif attend des mots-clés délimités, et les fichiers doivent rester lisibles par un humain (pas de tokeniseur greedy côté natif). **Reste** (optionnel) : niveaux float plus petits (binary16 / virgule fixe) pour coords bornées ; `ATN`, `RND` ; vars chaîne + `POKE` pour les listings restants. Investiguer aussi le **bug `IF (X AND 7)=0` qui fige l'interpréteur** via le tokeniseur (le natif est correct).
